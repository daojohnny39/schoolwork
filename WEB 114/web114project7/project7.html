Answer I

The first one is using curly braces when writing functions, if statements, and loops.

if (n < 0)
   alert(`Power ${n} is not supported`);

This kind of syntax can bring a lot of errors in the future when the writer decides to add more lines of code. Let's look at the "correct" syntax for that statement:

if (n < 0) {    // <-- opening bracket
   alert(`Power ${n} is not supported`);
}   // <-- closing bracket

Adding curly braces makes sure that the code will run accordingly to how the writer wants it to run. It also makes it easier to distinguish the scope of the code whether it's a function, if statement, or loop.

Answer II

Another practice commonly used is splitting long lines of code or text into separate lines to improve readability. Let's look at an example:

let str = ` ECMA International's TC39 is a group of JavaScript developers, implementers, academics, and more, collaborating with the community to maintain and evolve the definition of JavaScript.
`;

As you can see, it's pretty hard to distinguish the text from the code. Let's look at an example where the text is split into separate lines:

let str = `   // <-- break line here
   ECMA International's TC39 is a group of JavaScript developers,
   implementers, academics, and more, collaborating with the community
   to maintain and evolve the definition of JavaScript.
`;

As you can see, that text was a lot easier to read as well as the code. This is a good practice to use when there are long lines of text or code. (Especially on those if statements with multiple conditions)

Answer III

Another thing that is good practice is adding enough whitespace between lines of code. Let's look at an example:

function pow(x, n) {
   let result = 1;
   for (let i = 0; i < n; i++) {
      result *= x;
   }
   return result;
}

This piece of code will run correctly. However, the issue comes when you are trying to debug the code or just reading it in general. Let's look at another example where sufficient whitespace is added:

function pow(x, n) {

   let result = 1;

   for (let i = 0; i < n; i++) {
      result *= x;
   }

   return result;

}

As you can see, the whitespace allows your eyes to distinguish the different parts within the code. This is good general practice to implement in your code to allow other people to read it easier.

4. Explain the difference between a primitive, an array, and an object and include an example of each.

Answer

The differences between these 3 different types of data elements are pretty subtle but very obvious as well. Once you see a little bit of code for each of those types of data elements, you'll be able to distinguish them pretty easily. Let's look at an example of each:

Primitive:

let userName = "Johnny";
let userAge = 21;
let userGen = "male";

Each of these variables are considered primitive data types. They are pretty easy to tell apart from the other data types. It is also important to keep in mind that these primitive data types store only a single piece of data. (Integers, strings, booleans, etc.)

Arrays:

let nameArray = ["Johnny", "Nhat", "Dao"];
let numArray = [1, 2, 3, 4, 5];

Each of these variables are considered arrays. The declarations for arrays are pretty similar to a normal variable declaration. The difference is that arrays can hold multiple pieces of data. This allows us to create lists of data.

Objects:

let person = {
   name: "Johnny",
   age: 21
   class: "WEB 114"
};

This is an object declaration. Objects are similar to arrays in the sense that they can hold multiple pieces of data. Where objects differ from both arrays and primitive data types is that they hold attributes and values for the object declared. You can also access these different attributes like this:

console.log(person.name);

This would output "Johnny" to the console.

5. Explain the phrase "Fractions divided by powers of 2 are exactly represented in the binary system" and what universal problem this is referring to in all programming languages.

Another way to think about this is to ask yourself these questions -- can all decimal integers (the numbering system we use that uses values of 0-9) be accurately represented by the binary (the numbering system the computer uses with values of 0 and 1)? Can all decimal fractions be accurately represented by the binary numbering system? Which ones can? Which ones cannot?

Answer

The phrase "Fractions divided by powers of 2 are exactly represented in the binary system" is referring to the fact that fractions that are divided by powers of 2 are exactly represented in the binary system. You may run into an issue when you are trying to divide a fraction by 3 or 5. This is because computers can only store data in 0's and 1's.

This is a universal problem in all programming languages because of the way computers store data. The computer can only store up to 16 digits of data. (At least in JavaScript. Other languages may vary I think) When you divide a fraction by 3 or 5, the computer will try to store the data but it will run out of space.

One way to get around this issue is to multiply the fraction into an integer, then multiply it back to a fraction/decimal.
